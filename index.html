Requisitos funcionales — aplicación Android (MVP)

El MVP para Android cubre las funcionalidades esenciales del flujo cliente-especialista. El panel de administración y la gestión de verificación están disponibles solo en la versión web.

El MVP Android se valida en Android (no incluye iOS en esta etapa). La edición avanzada del perfil profesional del especialista queda disponible únicamente en web.

Requisitos no funcionales

Atributos de calidad del sistema Mi Chamba. Aplican tanto a la plataforma web como a la aplicación Android salvo indicación contraria.

IDCategoríaNombreDescripciónMétrica / criterio de aceptaciónPrioridad
Rendimiento
RNF-01RendimientoTiempo de respuesta de la APILas llamadas a la API REST deben responder en condiciones normales de carga.P95 ≤ 500 ms para endpoints de lectura; ≤ 1000 ms para escritura.Alta
RNF-02RendimientoLatencia del chatLos mensajes enviados por WebSocket deben entregarse al destinatario con baja latencia.Latencia extremo a extremo ≤ 300 ms en condiciones normales de red.Alta
RNF-03RendimientoCarga de la página webLa plataforma web debe cargarse rápidamente en dispositivos de gama media con conexión móvil.First Contentful Paint ≤ 2 s. Largest Contentful Paint ≤ 3 s.Alta
RNF-04RendimientoUsuarios concurrentesEl sistema debe soportar un número razonable de usuarios activos simultáneamente en el MVP sin degradación notable.Sin degradación medible con hasta 200 usuarios concurrentes.Media
RNF-05RendimientoPaginación de feedsLos feeds de pedidos y listas extensas deben paginar resultados para no sobrecargar la red ni el cliente.Máximo 20 ítems por página. Cursor-based pagination en la API.Media
Seguridad
RNF-06SeguridadCifrado de contraseñasLas contraseñas de usuario no deben almacenarse en texto plano en ningún caso.Hash con bcrypt, factor de costo mínimo 12. Sin almacenamiento de contraseña en claro.Alta
RNF-07SeguridadTransporte cifradoToda comunicación entre clientes y servidor debe realizarse sobre conexiones cifradas.HTTPS obligatorio en todos los endpoints. HTTP redirige a HTTPS. WebSocket sobre WSS.Alta
RNF-08SeguridadGestión de tokens JWTLos tokens de acceso deben tener vida corta. Los refresh tokens deben rotarse al usarse.Access token: expiración ≤ 15 min. Refresh token: expiración ≤ 7 días con rotación.Alta
RNF-09SeguridadAutorización por rolLos endpoints de la API deben verificar el rol del usuario autenticado antes de ejecutar cualquier acción.Guards de NestJS aplicados a todos los controladores. Tests de acceso cruzado entre roles.Alta
RNF-10SeguridadProtección contra XSSEl sistema debe sanitizar todas las entradas de usuario antes de almacenarlas o renderizarlas.Salida de datos de usuario siempre escapada. Sin innerHTML con datos sin sanitizar.Alta
RNF-11SeguridadProtección contra inyección SQLNo deben construirse queries SQL concatenando strings de entrada del usuario.Todas las consultas a la base de datos a través de Prisma (queries parametrizadas). Cero SQL crudo con interpolación de usuario.Alta
RNF-12SeguridadRate limitingLos endpoints de autenticación y publicación deben limitar la cantidad de solicitudes por IP para prevenir ataques de fuerza bruta y spam.Máximo 10 intentos de login por IP por minuto. Máximo 20 publicaciones de pedido por usuario por hora.Media
RNF-13SeguridadValidación de archivos subidosLos documentos subidos para verificación deben validarse en tipo y tamaño antes de almacenarse.Tipos permitidos: PDF, JPG, PNG. Tamaño máximo: 5 MB por archivo.Media
Usabilidad
RNF-14UsabilidadDiseño responsive webLa plataforma web debe funcionar correctamente en resoluciones de escritorio, tablet y móvil.Sin scroll horizontal ni elementos superpuestos en viewports desde 375 px hasta 1920 px.Alta
RNF-15UsabilidadCompatibilidad de navegadoresLa web debe funcionar correctamente en los navegadores de mayor uso.Chrome, Firefox y Safari en sus dos últimas versiones principales. Edge en su última versión.Alta
RNF-16UsabilidadMensajes de error comprensiblesLos mensajes de error del sistema deben ser claros para el usuario final, sin exponer detalles técnicos internos.Ningún stack trace ni mensaje de base de datos visible en la interfaz de usuario.Alta
RNF-17UsabilidadFeedback de accionesEl sistema debe confirmar visualmente al usuario el resultado de cada acción relevante (publicar pedido, enviar propuesta, etc.).Toda acción con efecto persistente muestra un mensaje de éxito o error en ≤ 1 s.Media
RNF-18UsabilidadAccesibilidad básicaLa interfaz web debe ser navegable con teclado y compatible con lectores de pantalla básicos.Todos los formularios tienen labels asociados. Imágenes tienen texto alternativo. Contraste de color ≥ 4.5:1 (WCAG AA).Baja
Disponibilidad y confiabilidad
RNF-19DisponibilidadUptime del servicioEl sistema debe estar disponible la mayor parte del tiempo en horario de uso típico.Disponibilidad ≥ 99 % mensual. Mantenimientos programados fuera del horario pico (00:00–06:00 ART).Alta
RNF-20DisponibilidadManejo de errores del servidorAnte fallos internos, el sistema debe responder con códigos HTTP apropiados sin exponer información sensible.Errores 5xx retornan JSON con mensaje genérico. Logging interno del error completo.Alta
RNF-21DisponibilidadBackups de la base de datosLos datos de usuarios, pedidos y transacciones deben respaldarse periódicamente.Backup automático diario del PostgreSQL administrado. Retención mínima de 7 días.Alta
RNF-22ConfiabilidadConsistencia de datosLas operaciones críticas (aceptar propuesta, cambiar estado de pedido) deben ser atómicas para evitar estados inconsistentes.Uso de transacciones de base de datos en operaciones que modifican múltiples tablas.Alta
Escalabilidad y mantenibilidad
RNF-23EscalabilidadEscalado horizontal del backendEl backend debe poder escalar agregando instancias sin cambios de arquitectura significativos.El servidor NestJS no guarda estado en memoria entre requests. Sesiones en Redis (post-MVP).Media
RNF-24EscalabilidadSeparación por módulosLa arquitectura del backend debe permitir modificar o reemplazar un módulo sin afectar a los demás.Cada dominio (auth, pedidos, propuestas, chat, etc.) en su propio módulo NestJS con interfaces definidas.Media
RNF-25MantenibilidadCobertura de testsEl código crítico del backend debe contar con tests automatizados para prevenir regresiones.Cobertura mínima del 60 % en servicios de dominio. Tests de integración para los flujos principales.Media
RNF-26MantenibilidadDocumentación de la APILa API REST debe estar documentada de forma que cualquier miembro del equipo pueda consumirla sin consultar el código fuente.Swagger generado automáticamente con @nestjs/swagger. Disponible en /api/docs en entorno de desarrollo.Media
RNF-27MantenibilidadTipado estrictoEl código TypeScript del proyecto debe ejecutarse sin errores de tipo en modo estricto.strict: true en tsconfig. CI rechaza builds con errores de tipo.Media
Privacidad y datos personales
RNF-28PrivacidadDatos mínimos requeridosEl sistema solo debe solicitar los datos personales estrictamente necesarios para el funcionamiento de la plataforma.Sin campos opcionales marcados como obligatorios. Sin recolección de datos de ubicación en tiempo real.Alta
RNF-29PrivacidadAcceso a datos propiosCada usuario solo puede acceder y modificar su propia información personal.Ningún endpoint expone datos de un usuario a otro sin relación activa (pedido compartido).Alta
RNF-30PrivacidadEliminación de cuentaEl usuario debe poder solicitar la eliminación de su cuenta y datos personales.Funcionalidad de baja de cuenta disponible. Datos personales anonimizados o eliminados en un plazo razonable.Baja
